Why index.html Contains No Data

This is the most common question. The answer is fundamental to understanding React. index.html is just an empty launch pad — React fills it at runtime using JavaScript.

Aspect Traditional HTML React (This Project)
index.html Contains ALL visible content Empty shell — just <div id="root">
Content appears Immediately on file load After JavaScript runs (milliseconds)
Data location Written manually in HTML In phases.ts / notes.ts — imported by React
Interactivity Needs manual DOM code Built into components with useState / useEffect
Dark mode, search Complex manual JavaScript Handled automatically by hooks and Tailwind

The 6-Step Boot Sequence

① Browser receives index.html

An almost-empty file. Contains only <div id="root"></div> and one script tag pointing to main.tsx.

② Browser reads the <script> tag

Loads /src/main.tsx — the JS bundle compiled by Vite from all your TypeScript files.

③ main.tsx executes

Calls ReactDOM.createRoot(document.getElementById('root')) — tells React where to work.

④ React renders App.tsx

App.tsx pulls in PHASES data, NOTES data, useTrackerState, useTheme, and all 8 components.

⑤ Components mount into #root

React injects all the generated HTML into the empty <div id="root">. The DOM is now full.

⑥ You see the full tracker ✅

Stats, phases, tasks, notes panel — everything rendered and interactive. index.html never changed.

Key insight: The index.html being empty is correct. It is the empty stage. React is the performer who fills it. This is called Client Side Rendering (CSR).

The Three Clean Layers

Like a well-designed n8n pipeline, the project has three distinct layers that never mix. Each layer only depends on the layer above it — never below.

LAYER 1 — DATA

Raw structured data. No logic, no React, no side effects.

Just TypeScript arrays and typed objects describing what the app contains. If you add a new day or fix a task title, only these files change.

phases.ts notes.ts
↓
LAYER 2 — LOGIC

Stateful hooks and pure utility functions. No JSX, no rendering.

Hooks own state and side effects (localStorage, DOM class). Utils are pure functions — same input always gives same output. Both are independently testable.

useTrackerState.ts useTheme.ts utils/index.ts
↓
LAYER 3 — UI

Pure presentation. Receives data as props, fires events upward.

Components never touch localStorage, never call fetch, never compute stats. They render what they receive. This makes them easy to test — pass props in, check what rendered out.

Header StatsBar PhaseCard DayBlock TaskRow NotesPanel
Senior dev rule: A UI component that reads from localStorage or calls fetch directly is a code smell. Keep layers clean — logic in hooks, display in components.

Full Data Flow — Like an n8n Pipeline

Flow 1 — App Startup

index.html loads
  └─ main.tsx executes (React bootstrap)
       └─ App.tsx renders
            ├─ useTrackerState() → reads localStorage → AppState{}
            ├─ useTheme() → reads localStorage → 'dark' | 'light'
            ├─ PHASES imported from phases.ts
            ├─ NOTES imported from notes.ts
            └─ All 8 components render with initial state

Flow 2 — User Ticks a Checkbox

TaskRow — user clicks the row
  └─ TaskRow.onToggle() fires
       └─ DayBlock passes up: onToggle(ti)
            └─ App.tsx handleToggle(pi, di, ti)
                 └─ useTrackerState.toggleTask(pi, di, ti)
                      ├─ state["p0d0t2"] = !state["p0d0t2"]
                      ├─ _streak + _lastDay updated if new day
                      ├─ saveState() → writes to localStorage
                      └─ React re-renders:
                           ├─ TaskRow — checkbox ticks ✓
                           ├─ StatsBar — Done++ Left--
                           └─ ProgressBar — bar fills further

Flow 3 — User Clicks the "📖 Notes" Button

DayBlock — user clicks Notes button for "Day 1"
  └─ DayBlock fires: onNotes("p0d0", "Day 1 — How The Web Works")
       └─ App.tsx handleOpenNotes(key, label)
            ├─ NOTES["p0d0"] looked up in notes.ts
            ├─ setActiveNote(noteData) — state changes
            └─ NotesPanel receives note prop
                 ├─ CSS: translateX(100%) → translateX(0) — slides in
                 ├─ Body scroll locked (overflow: hidden)
                 └─ Each NoteSection rendered by type:
                      h2 → heading, code → pre block, tip → callout...

Flow 4 — User Types in the Search Box

SearchBar input event fires
  └─ 200ms debounce timeout (prevents lag)
       └─ App.tsx setSearch("flexbox")
            └─ visiblePhases recomputed
                 └─ Each PhaseCard renders
                      └─ Each DayBlock checks visibility:
                           task.t.includes("flexbox")
                           || day.label.includes("flexbox")
                           || day.day.includes("flexbox")
                                ├─ Match → day renders normally
                                └─ No match → returns null (removed from DOM)

Flow 5 — Theme Toggle

Header — user clicks ◑ theme button
  └─ Header fires: onToggle()
       └─ App.tsx triggers: toggleTheme()
            └─ useTheme.ts
                 ├─ setThemeState('light' or 'dark')
                 ├─ document.documentElement.classList toggle 'dark'
                 └─ localStorage.setItem('devmap_theme', theme)
                      └─ Tailwind dark: variants activate/deactivate
                         All dark:bg-*, dark:text-* classes switch instantly

TypeScript Type System

All types live in src/types/index.ts and are imported by every file. They are the contracts that prevent data mismatches between layers.

Type / Interface Shape Used By
TaskKind 'concept' | 'code' | 'quiz' | 'project' phases.ts, utils, TaskRow badge
Task { t: string; k: TaskKind } phases.ts, DayBlock, TaskRow
DayData { day, label, noteKey, tasks: Task[] } phases.ts, DayBlock, PhaseCard
PhaseConfig { title, days, icon, color, dim, data: DayData[] } phases.ts, App.tsx, PhaseCard
NoteSection Union of 8 shapes: h2 | h3 | p | code | tip | warn | table | keypoints | quiz notes.ts, NotesPanel renderer
NoteData { phase: string; day: string; sections: NoteSection[] } notes.ts, NotesPanel, App.tsx
NotesMap Record<string, NoteData> notes.ts export, App.tsx lookup
AppState { [taskId]: boolean; _streak?: number; _lastDay?: string } useTrackerState, utils, all tests
Why strict: true matters: With "strict": true in tsconfig.json, TypeScript enables strictNullChecks, noImplicitAny, and 6 other checks. If you mistype a prop name, pass the wrong type, or forget to handle null, the build fails before deployment. This catches real bugs automatically.

How taskId() uniquely identifies every task

// Every task has a unique string key composed of:
// p = phase index,  d = day index,  t = task index

taskId(0, 0, 0)  → "p0d0t0"  // Phase 1, Day 1, Task 1
taskId(0, 0, 2)  → "p0d0t2"  // Phase 1, Day 1, Task 3
taskId(2, 4, 1)  → "p2d4t1"  // Phase 3, Day 5, Task 2

// AppState stores them as:
{
  "p0d0t0": true,   // checked
  "p0d0t2": false,  // unchecked
  "_streak": 5,
  "_lastDay": "Mon May 25 2026"
}

All npm Commands

Command When to Use What It Does
npm run dev Daily development Starts Vite dev server at localhost:5173. Hot module reload — changes appear instantly without page refresh.
npm run build Before deploying Runs tsc (TypeScript check) then vite build. Outputs optimised JS + CSS + HTML to dist/ folder.
npm run preview Test production Serves the dist/ folder locally. Simulates what Vercel will serve. Catches build-only bugs.
npm test CI / after changes Runs all 56 Vitest tests once and exits. Zero failures required to pass.
npm run test:watch Active development Re-runs tests on every file save. Instant feedback loop — see passes/fails as you code.
npm run test:ui Debugging tests Opens Vitest UI in browser — visual test runner with pass/fail details and stack traces.
npm run typecheck Quick verification Runs tsc --noEmit only. Faster than full build. Zero errors = types are clean.
npm run lint Before committing Same as typecheck + prints "No lint errors". Use before every git commit.
On iPad: None of these commands work on iPadOS — there is no Node.js runtime. Run them on a computer, or use StackBlitz (cloud browser) which runs them for you automatically.

Key Concepts — Plain English

Click any card to see the full explanation.

⚡

What is Vite?

The build tool that powers npm run dev and npm run build.
Vite is both a development server and a build tool. In development it serves your files instantly and hot-reloads changes in milliseconds (changes appear without even refreshing the page). In build mode it bundles all TypeScript + CSS + React into tiny optimised files that any server can host. It's much faster than the old Create React App tool — starts in under 300ms.
🧪

What is Vitest?

The test runner — runs your 56 tests with npm test.
Vitest is a testing framework built on top of Vite. It runs all your *.test.ts files, simulates a browser environment using jsdom, and gives you describe/it/expect/vi as global functions (no import needed). It's compatible with Jest so most testing tutorials apply. npm test runs all tests once; npm run test:watch re-runs on every file save for instant feedback.
🎨

What is Tailwind CSS?

Utility-first styling — classes instead of CSS files.
Instead of writing CSS files, Tailwind gives you utility classes applied directly in JSX. className="text-[13px] text-emerald-400 border border-emerald-500/20 rounded-lg" is all the styling needed. Tailwind's build step scans your source files and only generates CSS for classes you actually used — so the final CSS bundle is tiny. Responsive prefixes (sm: md: lg:) and state variants (hover: focus: dark:) are built in.
🪝

What is a Custom Hook?

A function starting with "use" that encapsulates reusable logic.
A custom hook is a regular TypeScript function whose name starts with "use" and which can call other hooks (useState, useEffect) inside it. useTrackerState() extracts all task state logic from App.tsx into its own testable file. The hook returns exactly what App.tsx needs (state, toggleTask, resetAll) and hides how localStorage works. This makes the logic independently testable with renderHook().
💾

What is localStorage?

Browser storage that survives page refreshes — no server needed.
localStorage is a key-value store built into every browser. It persists data across page refreshes, browser restarts, and even device reboots — but only for that browser on that device. Your tracker uses it to save which tasks are checked, the current streak count, and the last active day. No server, no database, no account needed. The limitation: if you open the app in a different browser or device, your progress doesn't carry over.
🌲

What is the DOM?

The live tree structure the browser builds from HTML.
When a browser reads HTML, it builds a tree of objects in memory called the Document Object Model. React creates a Virtual DOM — a JavaScript copy of this tree — and diffs it against the real DOM. When state changes, React only updates the parts that actually changed instead of rewriting everything. This is why React is fast even with many interactive elements. JavaScript can read and modify the DOM via document.querySelector() and friends.
📦

What is TypeScript strict mode?

strict: true in tsconfig enables all safety checks.
strict: true in tsconfig.json enables 7 strict checks at once: strictNullChecks (can't use a value that might be null), noImplicitAny (can't have untyped variables), strictFunctionTypes, strictBindCallApply, strictPropertyInitialization, noImplicitThis, and alwaysStrict. Without it TypeScript is lenient and misses real bugs. With it, if you mistype a prop name, pass the wrong type, or forget to handle undefined, the build refuses to compile.
🔄

What is Hot Module Reload?

Changes appear in the browser without refreshing the page.
When you run npm run dev and save a file, Vite detects the change and sends only the updated module to the browser. The browser swaps it in without losing your current state — your checked tasks, open panels, and typed search all remain. This makes development much faster than full page reloads. TypeScript errors also appear inline in the terminal instantly, so you catch mistakes as you type.
